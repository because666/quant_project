# 基于机器学习的量化投资选股策略设计及应用中期报告
## 一、项目实际进展情况、实施初步成效
### 1. 项目整体架构搭建完成
本项目“基于机器学习的量化投资选股策略设计及应用”目前已顺利完成核心架构的设计与搭建工作。项目采用模块化+分层化设计理念，构建了**数据层、算法层、策略层、回测层、应用层**五层架构，落地包含数据模块、策略模块、回测模块、主应用模块和前端界面在内的五大核心组件，形成了完整的量化交易系统框架。

整体架构严格遵循“配置→数据模块→策略模块→回测模块→主程序与前端界面”的单向数据流转逻辑，制定《项目接口开发规范V1.0》，统一模块间数据交互标准：所有跨模块数据采用Pandas DataFrame格式，时间列统一为datetime格式，股票代码统一为6位数字字符串，因子列命名采用英文小写+下划线格式，确保各环节衔接顺畅、高效协同，无数据格式不兼容风险。

### 2. 数据模块建设进展
数据模块已完成数据获取、数据清洗、因子挖掘三个核心组件的框架设计，建立了标准化的“获取-清洗-存储-校验”全流程数据处理机制，成功实现从原始数据到标准化因子数据的自动处理流程。

具体落地进展如下：
- **数据获取**：完成Tushare、AKShare、baostock三大数据源的接口适配，可支持沪深A股全市场股票日线行情、财务指标、行业分类数据的批量获取，内置请求频率限制（每秒2次）、断点续传、超时重试机制，避免接口封禁。已完成2018年1月1日-2025年12月31日沪深300+中证500共800只核心成分股的全量数据获取，生成raw_data.parquet原始数据文件，文件大小2.3GB，包含120万+行行情数据，数据完整率达98.7%。
- **数据清洗**：落地全流程数据预处理逻辑，缺失值处理采用“停牌日前向填充（最多5个交易日）+财务数据行业均值填充”双规则；异常值处理采用“3σ原则去极值+MAD中位数绝对偏差法二次校验”；数据标准化采用横截面Z-score标准化，消除因子量纲影响，最终生成standard_data.parquet标准化数据文件。
- **因子挖掘**：设计完成3大类共42个基础因子的自动化计算框架，包括18个技术指标类因子（MA均线、MACD、RSI、ATR波动率等）、12个量价类因子（区间收益率、波动率、价量相关性等）、12个基本面类因子（PE、PB、ROE、净利润增长率等），支持因子行业中性化、正交化处理，为策略研发提供标准化特征输入。

### 3. 策略模块研发进展
策略模块已完成模型训练、因子优化、策略引擎三个核心模块的架构设计，搭建了支持多种机器学习算法的时序滚动训练框架，从根源上避免未来函数，已初步完成模型训练与交易信号生成逻辑落地。

具体落地进展如下：
- **模型训练框架**：完成逻辑回归、随机森林、XGBoost三种主流算法的封装，内置6个月训练窗口+1个月验证窗口的时序滚动训练模式，训练集、验证集、测试集按7:2:1时序划分，不做随机打乱；支持超参数网格搜索、5折时序交叉验证与早停机制，防止模型过拟合。已完成基于2018-2022年训练数据的模型初步训练，生成trained_model.pkl模型文件，验证集AUC值达0.62，具备初步的收益预测能力。
- **因子优化模块**：实现因子有效性检验框架，可自动计算因子IC值、ICIR值，完成单因子回测与皮尔逊相关性分析，采用递归特征消除法（RFE）+特征重要性排序进行因子筛选，可自动剔除相关性大于0.7的冗余因子，保留对收益预测贡献度最高的核心因子，已输出初步的因子重要性排序报告。
- **策略引擎**：设计了完善的交易信号生成与过滤机制，明确标签定义为“个股未来20个交易日相对沪深300的超额收益率，横截面排名前20%为正样本（标签1），后20%为负样本（标签0）”；设置分层信号阈值：预测上涨概率≥65%生成买入信号（1），≤35%生成卖出信号（-1），中间区间生成持有信号（0）；同时内置ST股、停牌股、次新股过滤规则，规避无效交易风险，可实现三类交易信号的标准化自动输出。

### 4. 回测模块建设进展
回测模块已完成回测引擎、绩效分析器、报告生成器三个核心组件的设计，1:1还原A股市场交易规则，建立了多维度的策略绩效评估体系，可实现策略效果的量化验证与可视化呈现。

具体落地进展如下：
- **回测引擎**：完全贴合A股T+1交易制度与涨跌停板限制，实现涨停无法买入、跌停无法卖出的订单撮合逻辑；内置真实交易成本参数，默认设置双边佣金万分之1.5、卖出单边印花税千分之1、固定滑点0.2%；支持自定义回测参数，默认初始资金100万元、月度调仓、单只股票持仓上限5%、单行业持仓上限20%、最大持仓30只股票，可精准模拟真实交易环境，输出每日持仓明细与资金曲线。已完成基准定投策略测试，回测结果与理论值偏差率低于0.5%，确保引擎计算准确性。
- **绩效分析器**：设计了包含收益类、风险类、风险调整后收益类三大维度20+项核心指标的综合评估体系，核心指标包括年化收益率、累计收益率、相对基准超额收益率、最大回撤、年化波动率、夏普比率、索提诺比率、卡玛比率、信息比率、阿尔法系数、贝塔系数等，明确各指标的标准化计算公式，其中夏普比率采用一年期定存1.5%作为无风险收益率，可全面量化策略的盈利能力与风险水平。
- **报告生成器**：基于Plotly实现可视化报告自动生成，支持输出HTML格式交互式报告与Excel格式明细数据，报告包含策略绩效概览、净值曲线对比、回撤分析、收益分布、持仓交易分析、风险指标明细六大核心模块，所有图表支持缩放、明细查看与下载，可直观、全面地呈现策略回测效果。

### 5. 配置与串联模块进展
配置与串联模块建立了统一的配置管理体系，完成了全流程主控制器的框架设计，成功打通从数据获取到报告生成的完整业务链路，同时落地了基础风险管理机制。

具体落地进展如下：
- **统一配置管理**：落地三大核心配置文件，实现参数与代码的完全解耦。其中project_config.py为全局常量配置，包含数据时间范围、股票池、回测基础参数、无风险利率等；strategy_params.yaml为策略超参数配置，包含模型类型、训练窗口、信号阈值、调仓周期、仓位规则等；api_config.yaml为数据源接口配置，包含接口token、请求频率、超时重试次数等，支持用户无需修改核心代码即可完成策略参数调整。
- **主流程控制器**：完成main_pipeline.py主流程框架设计，内置7个标准化执行步骤：加载配置→数据获取清洗→因子计算预处理→模型训练预测→交易信号生成→回测模拟交易→报告生成输出，实现全流程一键执行；同时内置断点续跑功能，若某一环节执行失败，可从失败节点重新运行，无需重复执行已完成步骤，大幅提升调试效率。
- **风险管理模块**：设计了四大类基础风控规则，包括个股止损规则（单只股票浮亏超10%强制平仓）、仓位风控规则（沪深300指数20日跌幅超8%，总仓位上限降至30%）、集中度风控（单行业持仓不超20%、单只股票持仓不超5%）、流动性风控（剔除日均成交额低于5000万元的个股），可实现策略运行过程中的风险实时监控，降低策略极端行情下的回撤风险。

## 三、存在问题及拟采取的解决措施
### 1. 核心算法实现待完善
**具体问题**：目前各模块仅完成主体框架搭建与基础功能开发，核心精细化算法代码尚未完全落地，导致模块核心能力未完全释放。具体包括：数据模块中基本面因子自动更新、因子正交化代码未完全实现；策略模块中超参数网格搜索、时序交叉验证的全自动化代码未完全落地，因子筛选功能未实现全流程闭环；回测模块中Brinson收益归因、交易成本敏感性分析代码尚未开发；主流程控制器中异常处理、日志分级记录功能未完全实现。

**拟采取的解决措施**：
1. 制定分模块代码开发计划表，明确各模块核心代码的开发优先级、责任人与完成时限：数据模块核心代码（第1周完成，责任人：数据组组长）、策略模块核心代码（第1-2周完成，责任人：策略组组长）、回测模块核心代码（第2-3周完成，责任人：回测组组长）、主流程控制器代码（第3周完成，责任人：系统集成组组长），确保开发工作按节点推进。
2. 采用AI辅助编程（GitHub Copilot+CodeGeeX）提升开发效率，要求所有核心函数必须包含功能说明、输入输出参数注释、异常处理逻辑，代码注释覆盖率不低于30%，保障代码的可读性与可维护性。
3. 建立代码双审机制，每段核心代码开发完成后，先由模块负责人初审，再由交叉小组复审，重点审核代码是否存在未来函数、逻辑漏洞、性能问题，审核通过后方可合并至Git主分支，确保代码的规范性与稳定性。
4. 搭建Git分支开发模式，每个模块建立独立开发分支，每周合并一次开发进度，同步解决开发过程中的问题，确保代码版本可控。

### 2. 模块间集成测试尚未开展
**具体问题**：各模块目前均处于独立开发与单元测试状态，尚未开展完整的端到端集成测试，存在多方面潜在风险：数据模块输出的标准化数据格式与策略模块要求的输入格式尚未完成兼容性校验；策略模块生成的交易信号格式与回测模块的订单输入规则尚未完全匹配，存在字段不统一、数据类型不一致的问题；未建立自动化测试体系，无法实现全流程自动化回归测试；未针对数据源中断、模型预测失败等极端异常场景进行测试，系统容错能力未经验证。

**拟采取的解决措施**：
1. 制定分阶段集成测试计划，第一阶段（核心代码完成后3个工作日内）完成两两模块间的接口联调，重点验证数据模块→策略模块、策略模块→回测模块、回测模块→报告生成模块的数据流兼容性，输出接口联调报告；第二阶段（第4周内）完成全流程端到端集成测试，实现从数据获取到报告生成的全流程一键跑通，无报错、无数据异常。
2. 编写标准化测试用例，覆盖正常流程、边界条件、异常场景三大类，测试用例总数不低于50条，其中异常场景用例占比不低于30%，明确每个测试用例的输入、预期输出与验收标准。
3. 基于Pytest搭建自动化测试框架，实现单元测试、集成测试的自动化执行，每次代码合并至主分支时自动触发测试，测试不通过无法合并，确保代码迭代不影响已有功能。
4. 组建专项测试小组，由系统集成组组长牵头，每个小组选派1名成员参与，负责测试用例编写、测试执行、问题记录与跟踪，建立BUG管理台账，明确BUG等级、修复时限、责任人，确保所有严重及以上等级BUG在24小时内完成修复。

### 3. 数据源稳定性风险
**具体问题**：项目当前依赖AKShare/Tushare等外部数据源，存在多方面稳定性风险：Tushare接口有调用频次限制，普通用户每分钟最多调用500次，批量获取全市场数据时易触发限制，导致数据获取失败；AKShare为网页爬虫接口，目标网站页面结构变更会导致接口失效，存在数据获取中断风险；未建立完善的本地数据备份机制，单次数据获取失败会导致后续流程全部中断；未对获取的数据进行完整性自动化校验，存在数据缺失但未被识别的情况，影响模型训练效果。

**拟采取的解决措施**：
1. 实现三级数据本地缓存机制，建立raw_data、standard_data、factor_data三级缓存目录，按年份+股票代码对数据文件进行分区存储，已获取的数据自动缓存至本地，后续流程优先读取本地缓存数据，仅当缓存数据缺失时才调用外部接口，将接口调用频次降低80%以上，减少对外部数据源的实时依赖。
2. 优化数据获取模块的异常处理逻辑，添加指数退避重试机制，接口调用失败后，分别等待1s、2s、4s、8s、16s后重试，最多重试5次；重试失败则记录错误日志，继续执行后续股票的数据获取，避免单只股票获取失败导致全流程中断。
3. 搭建多数据源冗余备份体系，确定“Tushare为主数据源，AKShare为第一备用数据源，baostock为第二备用数据源”的组合，编写多数据源适配层，统一不同数据源的输出字段格式，当主数据源调用失败时，自动切换至备用数据源，保障数据供应的连续性。
4. 建立数据完整性自动化校验机制，数据获取完成后，自动校验数据的时间连续性、字段完整性、股票数量覆盖率，设置数据完整率阈值95%，低于阈值时自动触发补充获取，同时生成数据质量报告，明确缺失数据的股票代码、时间区间与缺失字段，便于人工复核。

### 4. 模型效果验证不足
**具体问题**：目前仅生成初步的预训练模型文件，尚未对模型效果进行多维度、全场景的回测验证，无法确定模型的选股有效性与泛化能力。具体包括：未开展多算法、多参数、多训练窗口的对照实验，无法确定最优模型配置；未开展严格的样本外测试，仅用训练集内数据完成验证，存在过拟合风险；未针对牛市、熊市、震荡市等不同市场行情进行分场景回测，无法确定策略在极端市场环境下的稳定性；未对交易信号的胜率、盈亏比进行有效性检验，无法明确信号的实际盈利贡献。

**拟采取的解决措施**：
1. 设计多组对照实验，开展模型横向对比测试，实验变量包括3种机器学习算法、4种训练窗口长度（3/6/12/24个月）、3种标签定义方式（未来5/10/20日超额收益），共36组实验，每组实验完成3次重复验证，以验证集AUC值、样本外年化收益率、夏普比率为核心评价指标，筛选出最优模型配置。
2. 开展严格的样本外测试，将全量数据划分为训练集（2018-2022年）、验证集（2023年）、样本外测试集（2024-2025年），模型训练过程完全不接触样本外测试集数据，仅用最优模型在样本外数据中进行回测，验证模型的泛化能力，验收标准为样本外测试集AUC值不低于0.58，相对沪深300的年化超额收益率不低于8%。
3. 开展分市场行情的场景化测试，将历史行情划分为牛市、熊市、震荡市三个场景，分别在不同场景中进行回测，统计策略在各场景下的年化收益率、最大回撤、胜率，确保策略在熊市中最大回撤不超过20%，震荡市中相对基准有稳定超额收益。
4. 建立交易信号有效性检验体系，统计买入信号的未来5/10/20日胜率、平均收益率、盈亏比，卖出信号的未来5/10/20日下跌概率、平均跌幅，剔除胜率低于50%的无效信号区间，优化信号生成阈值，提升交易信号的盈利贡献度。
5. 开展过拟合专项检验，对比模型在训练集、验证集、样本外测试集的绩效表现，若训练集年化收益率超过样本外测试集2倍以上，判定为存在过拟合，通过降低模型复杂度、增加正则化强度、减少特征数量等方式优化模型，消除过拟合风险。

### 5. 前端界面开发滞后
**具体问题**：前端界面目前仅完成框架设计与功能规划，核心交互功能尚未实现代码开发，无法直观呈现项目成果与策略运行效果。具体包括：核心的参数配置、数据可视化、结果展示功能均未落地，无法实现与后端模块的交互；未完成前后端接口联调，无法实现前端参数修改实时传递至后端，后端计算结果实时回显至前端；界面交互逻辑未细化，未考虑异常场景下的前端提示与容错处理；可视化图表类型单一，无法满足策略效果多维度展示的需求。

**拟采取的解决措施**：
1. 采用Streamlit轻量级框架分阶段实现前端功能开发，第一阶段（第3-4周）完成MVP最小可行产品开发，实现策略参数可视化配置、回测核心结果展示、回测报告在线查看与下载三大核心功能；第二阶段（第5-6周）完成全功能开发，补充数据预览、因子分析、模型训练结果展示、交易信号明细查询等功能。
2. 完成前后端接口封装，基于Streamlit的session_state实现前端与后端主流程的状态同步，前端参数修改后自动更新配置文件，触发后端流程执行，后端计算完成后自动将结果数据、图表渲染至前端页面，实现全流程可视化操作。
3. 优化界面交互逻辑，设计“配置参数→运行策略→查看结果”的标准化用户操作流程，每个步骤添加操作指引与提示信息，针对参数输入错误、流程运行失败等异常场景，添加友好的错误提示与解决方案建议，提升用户操作体验。
4. 丰富可视化展示内容，基于Plotly实现交互式图表，包括净值与基准对比曲线、回撤动态曲线、月度收益热力图、因子相关性热力图、因子重要性条形图、持仓行业分布饼图等，所有图表支持缩放、悬停查看明细、下载等交互功能。
5. 安排专人专职负责前端开发，明确开发进度节点，每周同步开发进度，确保前端开发与后端模块开发同步完成，第4周实现前后端全流程联调成功，第6周完成全功能界面交付。
