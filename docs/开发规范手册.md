# 项目开发规范手册

> 项目名称：基于排序学习的量化投资选股策略设计与应用  
> 适用范围：`backend/`、`frontend/`、`docs/` 全仓库

## 1. 代码风格规范

### 1.1 Python（后端）

- 遵循 `PEP8` 与 `PEP257`（文档字符串）。
- 使用 4 空格缩进，禁止使用 Tab。
- 单行长度建议不超过 88（与常见格式化工具兼容）。
- 导入分组顺序：标准库 -> 第三方库 -> 本地模块。
- 公共函数、核心类必须提供简要 docstring（说明输入/输出/副作用）。
- 类型注解要求：
  - 新增后端模块必须补全函数签名类型注解。
  - 对外接口（API、数据库层）返回值类型必须明确。

### 1.2 TypeScript（前端）

- 使用 `ESLint + Prettier` 作为唯一代码格式与静态检查标准。
- 保持统一风格：
  - 使用单引号；
  - 语句末尾不加分号（遵循当前 Prettier 配置）；
  - 尾随逗号开启（`trailingComma: all`）。
- TypeScript 约束：
  - 避免 `any`，优先使用明确类型或泛型；
  - API 响应类型必须在 `src/types` 定义。

### 1.3 命名规范

- 变量/函数：`snake_case`（Python）、`camelCase`（TS）。
- 类名：`PascalCase`（如 `DeepSeekClient`）。
- 常量：`UPPER_SNAKE_CASE`（如 `API_TIMEOUT_SECONDS`）。
- 文件命名：
  - Python 模块：`snake_case.py`
  - React 组件：`PascalCase.tsx`
  - 工具与服务模块：`camelCase.ts` 或 `snake_case.py`（按语言生态）
- 禁止拼音缩写和无语义命名（如 `tmp1`, `data2`）。

## 2. 目录结构规范

### 2.1 根目录

- `backend/`：后端服务与量化策略核心逻辑
- `frontend/`：前端网站
- `docs/`：文档与规范
- `README.md`：项目总览与快速启动

### 2.2 后端目录（当前基线）

- `backend/src/main.py`：FastAPI 应用入口（路由挂载、中间件、异常处理）
- `backend/src/api/v1/`：版本化 API 路由
- `backend/src/database/`：ORM 模型、会话管理、数据库初始化逻辑
- `backend/src/utils/`：日志、通用工具
- `backend/src/schemas/`：请求/响应数据模型
- `backend/data/`：本地数据（数据库、parquet 缓存）
- `backend/models/`：模型文件（后续阶段）
- `backend/logs/`：运行日志

### 2.3 前端目录（当前基线）

- `frontend/src/components/`：可复用组件
- `frontend/src/pages/`：页面级组件
- `frontend/src/services/`：API 请求与数据服务
- `frontend/src/types/`：类型定义
- `frontend/src/hooks/`：自定义 hooks
- `frontend/src/utils/`：工具函数
- `frontend/src/styles/`：全局样式与主题变量

## 3. Git 提交规范

- 使用 `Conventional Commits`：
  - `feat:` 新功能
  - `fix:` 缺陷修复
  - `refactor:` 重构（不改变行为）
  - `docs:` 文档更新
  - `test:` 测试相关
  - `chore:` 构建/依赖/脚本
- 提交信息格式：
  - `type(scope): subject`
  - 示例：`feat(backend): add deepseek streaming client`
- 一次提交只做一类变更，避免“混合提交”。
- 提交前必须通过本地基础检查（至少可启动、无明显语法错误）。

## 4. 接口设计规范

### 4.1 RESTful 约定

- URL 使用名词复数与小写短横线风格，例如：`/api/v1/backtest-results`
- 版本前缀统一：`/api/v1`
- 常用方法语义：
  - `GET`：查询
  - `POST`：创建/触发任务
  - `PUT/PATCH`：更新
  - `DELETE`：删除

### 4.2 请求与响应

- 请求体使用 JSON。
- 响应统一结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

- 失败响应示例：

```json
{
  "code": 500,
  "message": "internal server error",
  "data": null
}
```

### 4.3 状态码约定

- `200`：成功
- `400`：参数错误
- `401/403`：未授权/无权限（如后续引入鉴权）
- `404`：资源不存在
- `422`：请求体校验失败
- `500`：服务器内部错误

## 5. 数据库设计规范

### 5.1 命名约定

- 表名：小写复数，下划线分隔（如 `backtest_results`）。
- 字段名：小写下划线（如 `created_at`）。
- 主键：统一 `id`（自增整型）。
- 时间字段：每张表必须包含 `created_at`、`updated_at`。

### 5.2 索引与约束

- 高频查询字段必须建立索引（如 `stock_code`、`trade_date`、`account_id`）。
- 业务唯一性必须使用唯一约束（如 `backtest_id`、`advice_id`）。
- 索引命名建议：`idx_<table>_<field>`。
- 唯一约束命名建议：`uq_<table>_<field>`。

### 5.3 JSON 字段使用

- 动态结构（如因子集合、持仓快照）允许使用 JSON 字段。
- JSON 字段必须在文档中定义 key 语义与示例，避免“隐式协议”。

## 6. UI 设计系统（占位，任务7补充）

> 本章节当前仅定义结构与命名规则，具体数值由任务7补充并冻结。

### 6.1 设计令牌（Design Tokens）

- 颜色变量（占位）：
  - `--color-primary`
  - `--color-success`
  - `--color-warning`
  - `--color-danger`
  - `--color-bg`
  - `--color-text`
- 字体变量（占位）：
  - `--font-family-base`
  - `--font-size-base`
  - `--font-size-h1/h2/h3`
- 间距变量（占位）：
  - `--space-1` ~ `--space-8`
- 圆角变量（占位）：
  - `--radius-sm/md/lg`
- 阴影变量（占位）：
  - `--shadow-sm/md/lg`

### 6.2 使用规范

- 所有新页面优先复用设计令牌，禁止硬编码零散色值。
- 组件样式优先在 `src/styles/` 统一维护。
- 图表（ECharts/Plotly）主题颜色必须与设计令牌保持一致。

## 7. 错误处理与日志规范

### 7.1 后端异常处理

- 业务代码必须捕获可预期异常并返回可读错误信息。
- 未捕获异常由全局异常处理器兜底，返回统一 JSON 结构。
- 禁止将堆栈信息直接暴露给前端。

### 7.2 日志规范

- 日志级别：
  - `INFO`：关键流程状态
  - `WARNING`：可恢复异常
  - `ERROR`：影响功能的错误
- 日志内容必须包含必要上下文（模块名、关键参数、错误原因）。
- 生产环境禁止打印敏感信息（密钥、账户隐私数据）。

## 8. 执行与维护要求

- 本手册是默认强约束，代码评审以本手册为准。
- 如需修改规范，需先更新本手册再实施代码变更。
- 每个阶段结束时复核一次规范执行情况并记录偏差。
